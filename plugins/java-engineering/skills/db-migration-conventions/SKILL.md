# Database Migration Conventions

Migrations run against databases WITH DATA in every environment except a fresh dev
machine. Write every migration as if the table has a million rows and zero downtime
is expected.

## Absolute rules

- Never edit an applied migration; always a new versioned file forward.
- Never `DROP COLUMN` or `DROP TABLE` carrying data without an explicit,
  user-approved data plan. State the data consequence in a comment at the top.
- `ADD COLUMN ... NOT NULL` FAILS on non-empty tables unless it has a `DEFAULT`.
  Pattern: add nullable (or with default) тЖТ backfill тЖТ then `SET NOT NULL`.
- Type changes: prefer `ALTER COLUMN ... TYPE new_type USING <cast expression>` over
  drop-and-re-add, which destroys data.
- Every migration must be safe to run on a NON-EMPTY database. If it is only valid
  for empty/dev databases, refuse to write it that way; do the expand-contract
  version instead, or explicitly tell the user it is dev-only and destructive.

## ExpandтАУcontract (the default strategy for breaking changes)

1. EXPAND: add the new column/table alongside the old (nullable/defaulted).
2. MIGRATE: backfill data (`UPDATE ... SET new = <derived from old>`), deploy code
   that writes both / reads new.
3. CONTRACT: only after code no longer touches the old shape, a LATER migration
   drops it. Never expand and contract in the same migration.

## Keys, identity, and constraints

- Replacing a primary key: new column + backfill + unique index + constraint swap тАФ
  not DROP PK / DROP COLUMN / ADD COLUMN.
- Auto-generated ids need their generation wired: `BIGINT GENERATED BY DEFAULT AS
  IDENTITY` (or a sequence) тАФ a bare `BIGINT` column will not populate itself, and
  ORMs (e.g. Panache's Long id) expect identity/sequence to exist.
- When dropping a constraint to alter something, re-create it (or its replacement)
  in the SAME migration; never leave a table without its PK/FK/unique guarantees
  "temporarily".
- Index creation on large tables: `CREATE INDEX CONCURRENTLY` (Postgres) in its own
  migration with `flyway.executeInTransaction=false` noted, or accept the lock and
  say so.

## Hygiene

- One logical change per migration; name describes intent:
  `V1_2_0__Add_dedup_key_to_notifications.sql`.
- Top-of-file comment: what, why, and data impact (including "none").
- Keep migrations in sync with the ORM mapping they serve тАФ if the entity says Long,
  the column is BIGINT identity; verify rather than assume.
- Down/undo migrations are not used; recovery is a new forward migration.
